What is RLHF Annotation Studio?

How human opinions teach AI to be better — and what this platform does to collect them

AI doesn't learn from rules — it learns from votes

Imagine you're training a dog. You don't hand it a rulebook. You reward the right behavior and ignore the wrong ones. Over thousands of small moments, the dog figures out what "good" looks like — from your feedback.

The world's most powerful AI models are trained the same way — except instead of treats, they get human preference data. Millions of tiny votes: "This response was better than that one." This process is called RLHF — Reinforcement Learning from Human Feedback.

💡
The Core Insight

Every time you pick "Response A is better than Response B," you're casting a vote that shapes how the next version of that AI thinks and talks. Annotation is how human judgment gets baked into machine intelligence.

This app is the factory floor for RLHF data

RLHF Annotation Studio is a full-stack platform where human annotators review AI outputs, rate them, compare them, and flag problems — all in a structured, trackable way that can be exported as training data.

⚖️

Comparison Tasks

Show two AI responses side-by-side. Pick which one is better. This creates preference pairs used directly in RLHF training.

⭐

Rating Tasks

Score a single AI response on quality, safety, helpfulness, or any custom dimension. Like giving a restaurant 4 out of 5 stars.

🏆

Ranking Tasks

Drag-and-drop multiple AI responses into order from best to worst. More nuanced than binary comparisons.

📦

Task Packs

Bundle annotation tasks into shareable packs. Assign them to teams, track completion, export results as training data.

Your journey: from login to annotation in 4 steps

Here's exactly what happens when you use this app — end to end:

1
Sign in with your credentials

The app creates a JWT token and stores it in a secure cookie. Every action you take is now tied to your identity and role.

2
Load a task pack

The frontend fetches your assigned tasks from the API and stores them in local state. You see a list of AI responses waiting to be judged.

3
Annotate: pick, rate, or rank

You make your judgment. The frontend tracks your decision in Zustand state (your browser's memory) and records how long you took.

4
Submit — data flows to the database

Your annotation travels from browser → FastAPI backend → PostgreSQL database. It becomes a permanent record, ready for quality review and export.

In production: two clouds, one app

When this app is deployed, it splits across two hosting providers — each playing to their strengths:

🪰

Fly.io — Backend + Database

The FastAPI server and PostgreSQL database run on Fly.io, a platform that runs Docker containers close to users. The backend config lives in backend/fly.toml.

🔗
Why Split Them?

Vercel is unbeatable for static pages and SSR. Fly.io is better for long-running Python servers. Using the right tool for each job is a common production pattern — you'll hear engineers call this "separation of concerns."

How Your Click Becomes Data

Auth, state management, and the request lifecycle — traced from browser to database

Authentication: the concert wristband model

At a concert, the bouncer checks your ID once at the door and gives you a wristband. After that, you can move anywhere in the venue without showing ID again — everyone just checks your wrist.

This app works exactly the same way. You show your password once (authentication). The server issues a JWT token (your wristband). Every subsequent request carries that token — the server just checks the wrist.

🔑
Two tokens, one identity

This app stores the JWT in two places: localStorage (for API calls) and an rlhf_session cookie (for page route protection). The cookie tells Next.js middleware "this person is logged in." The JWT tells FastAPI "this person is this specific user with this role."

The code that guards every page

This file runs before every page load. If you don't have the session cookie, you get bounced to the login page — you never even see the page content.

CODE — frontend/src/middleware.ts 
const PROTECTED_PREFIXES = [
  "/dashboard",
  "/analytics",
  "/reviews",
  "/task",
  "/exams",
  // ... 10 more routes
];

const hasSessionCookie =
  Boolean(request.cookies
    .get("rlhf_session")?.value);

if (hasSessionCookie) {
  return NextResponse.next();
}
url.pathname = "/auth";
return NextResponse.redirect(url);
PLAIN ENGLISH

Here's a list of pages that require login. If you try to visit any of these without being logged in, you get redirected.

 

Check if the browser is carrying the session cookie. The ?.value safely handles the case where the cookie doesn't exist at all.

 

If the cookie exists: let them through. Show the page normally.

If not: change the destination URL to the login page and redirect. They never see the protected content.

🗺️
Why is this in middleware, not in each page?

If you checked for login inside each individual page, you'd have to copy that code 15+ times. Middleware runs once for all routes — it's a single chokepoint. Engineers call this the "single responsibility" principle: one place handles one job.

Your workspace in memory: Zustand state

Think of Zustand like a shared whiteboard in a conference room. Every component in the app can look at it, write on it, and erase from it. Crucially, this whiteboard has a memory — it survives page refreshes by saving itself to localStorage.

Here's what the annotation workspace stores at any given moment:

user + token Who you are and your JWT — sent with every API request
tasks[] Your loaded task pack — the AI responses waiting to be judged
annotations{} Your choices so far, keyed by task ID — won't disappear on refresh
taskTimes{} How long you spent on each task — used for quality metrics
currentTaskIndex Which task you're on right now — so "back" always works

The whiteboard setup — line by line

CODE — frontend/src/lib/state/store.ts 
export const useAppStore =
  create<AppState>()(
  persist(
    (set, get) => ({
      user: null,
      token: null,
      tasks: [],
      annotations: {},

      setAuth: ({ user, token }) => {
        localStorage.setItem(
          "rlhf_authToken", token);
        document.cookie =
          "rlhf_session=1; Path=/;";
        set({ user, token });
      },
    })
  )
);
PLAIN ENGLISH

Create a shared data store using Zustand. persist wraps it so it automatically saves to localStorage.

 

Starting values: no user logged in, empty task list, no annotations yet.

 

setAuth is called after login succeeds. It does three things at once:

① Saves the JWT to localStorage so future API calls can include it.

② Sets the session cookie so the middleware knows you're logged in.

③ Updates the store's user and token values so every component knows who's logged in.

The Clever Engineering

Four patterns that make this codebase production-grade — and how to request them from AI

Pattern 1 — Async: the air traffic controller model

A traditional web server is like a pharmacist who can only help one customer at a time. While they're filling one prescription, everyone else waits in line. For a few users, fine. For hundreds? Disaster.

FastAPI uses async/await — more like an air traffic controller. They can manage dozens of planes simultaneously: "Runway 1, hold. Runway 2, cleared to land. Runway 3, begin approach." No plane waits for another to land before getting instructions.

CODE — backend/app/routers/tasks.py (pattern) 
@router.post("/tasks/{task_id}/annotate")
async def submit_annotation(
    task_id: int,
    body: AnnotationCreate,
    db: AsyncSession = Depends(get_db),
    current_user: User = Depends(get_current_user),
):
    return await save_annotation(
        db, task_id, body, current_user
    )
PLAIN ENGLISH

When a POST request arrives at /tasks/123/annotate, run this function.

async def — this function can pause while waiting for the database, handling other requests in the meantime.

body: AnnotationCreate — FastAPI automatically validates the request body matches this shape. If it doesn't, it returns a 422 error before the function even runs.

Depends(get_db) — automatically inject a database connection. Depends(get_current_user) — automatically verify the JWT and inject the logged-in user. Both are "middleware injected" into the function.

await — pause here while the database does its work. Other requests run during this wait.

Pattern 2 — Pydantic: the shape police

Imagine ordering from a form where every field is validated before you can submit. Pydantic is FastAPI's built-in form validator — but for JSON data coming into the API.

🛡️
Why This Matters for You as a Vibe Coder

When you tell AI "add a new field to the annotation form," you also need to say "add it to the Pydantic schema in backend/app/schemas/." If you don't, FastAPI will silently drop that field — or crash with a validation error. This is the #1 "why isn't my new field saving?" gotcha.

schemas/ Pydantic models that define what data can enter and exit each API endpoint
models/ SQLAlchemy models that define how data is stored in PostgreSQL tables
422 error Validation failed — the request body didn't match the Pydantic schema

A new field needs to be added in both places: the schema (for validation) and the model (for storage). AI sometimes forgets one.

Pattern 3 — The LLM Judge: AI evaluating AI

One of the most powerful features: the platform can run an LLM to automatically evaluate annotation quality — comparing what a human annotator chose against what the AI model thinks is correct. This is called "LLM-as-a-judge."

1

Annotator submits a rating for an AI response

→
2

Judge service sends the response + annotation to an LLM with a rubric prompt

→
3

LLM returns its own score and reasoning

→
4

Platform compares: human vs LLM. Flags large disagreements for review.

⚖️
Override, Accept, Reject

When the LLM judge disagrees with an annotator, a reviewer can accept the LLM's assessment, reject it, or override it with a manual decision. The platform tracks all three paths — giving you a complete audit trail of every quality decision.

Pattern 4 — Alembic: version control for your database

Your code lives in Git — every change is tracked and reversible. But databases are different. If you add a column to a table, how do you make sure every deployment gets that column? That's what Alembic does.

Think of it like sheet music for an orchestra. Every instrument player (every server running your code) follows the same score. If you add a new note, everyone gets the updated score.

✍️

Generate migration

alembic revision --autogenerate -m "add_score_column" — Alembic inspects your models and writes the migration script automatically.

▶️

Apply to database

alembic upgrade head — runs all pending migrations in order. Safe to run multiple times — already-applied migrations are skipped.

⏪

Roll back

alembic downgrade -1 — undo the last migration. Like Ctrl+Z for your database schema. Critical when a deployment goes wrong.

⚠️
The AI Gotcha

When you ask AI to "add a new field to the database," it will update the SQLAlchemy model — but may forget to generate the Alembic migration. Always follow up with: "now generate the Alembic migration for this change." Without it, the column exists in code but not in the actual database.

Debug Like an Engineer

The most common failure modes, how to find them, and how to steer AI out of stuck loops

The usual suspects: what breaks most often

Every full-stack app has a predictable set of failure modes. Knowing these by name lets you tell AI exactly where to look — instead of watching it spin in circles for 10 minutes.

🚫
CORS errors

CORS fires when the frontend domain isn't in FastAPI's allowed origins list. Symptom: "Network Error" in browser console, request never reaches the backend. Fix: add your Vercel URL to allow_origins in backend/app/main.py.

🔐
JWT expiry — "suddenly logged out"

JWTs expire after 30 days. When they do, every API call returns 401. Symptom: user seems logged in (cookie exists) but all data requests fail. Fix: implement token refresh logic, or increase the expiry window for development.

🔌
Database connection pool exhausted

Under load, if too many async connections are held open, new requests time out. Symptom: 503 errors during traffic spikes. Fix: tune pool_size and max_overflow in the SQLAlchemy engine config.

📦
Stale Zustand state

Sometimes old annotation data persists in localStorage across sessions. Symptom: user sees tasks from a previous session, or their old answers auto-fill new tasks. Fix: call clearAnnotations() on new session start, or add a version key to the persist config.

🔄
Missing Alembic migration

AI adds a column to the model but forgets to migrate. Symptom: 500 errors when saving, with "column does not exist" in server logs. Fix: alembic revision --autogenerate -m "fix" then alembic upgrade head.

Spot the bug: something is wrong in this route

This FastAPI route is supposed to save an annotation and return the saved record. One line contains a critical mistake. Click the line you think is wrong:

Find the bug in this annotation endpoint:

1 @router.post("/tasks/{task_id}/annotate")
2 async def submit_annotation(task_id: int, body: AnnotationCreate,
3 current_user = Depends(get_current_user), db = Depends(get_db)):
4 annotation = Annotation(task_id=task_id, **body.dict())
5 db.add(annotation)
6 return annotation
💡
The Async DB Gotcha

With async SQLAlchemy, you must await db.commit() to actually save to the database, and await db.refresh(annotation) to reload the record (with its database-assigned ID) from the DB. AI frequently writes the db.add() but forgets the commit. This is the #1 "data is not saving" bug in async FastAPI code.

Getting unstuck: break AI out of bug loops

AI gets into "debug loops" — trying the same fix 3 different ways when it doesn't understand the root cause. Here's how to break the loop using what you now know:

1
Name the layer, not just the symptom

Instead of "annotations aren't saving," say "the FastAPI route returns 200 but PostgreSQL never gets the write. I think the Alembic migration is missing or the await db.commit() is absent." This tells AI exactly where to look.

2
Read the server logs, not just the browser

The browser shows you the HTTP status code. The real error is in the FastAPI server logs (or Fly.io logs). Give AI the actual Python stack trace — it narrows the problem from 10 possibilities to 1.

3
Check Zustand state in browser devtools

Open browser devtools → Application → Local Storage → look for the Zustand key. If the annotation data is there, the problem is in submission (network/backend). If it's not there, the problem is in the UI (state management).

4
Isolate with a direct API call

Use the FastAPI automatic docs at /docs (built-in Swagger UI) to call the endpoint directly without the frontend. If it works there, the bug is in Next.js. If it fails there, the bug is in FastAPI or the database.

What to build next: the feature map

Now that you understand the architecture, here's how to think about extending it. Each feature lives in a predictable place:

📊

New annotation type

New Pydantic schema → new SQLAlchemy model → Alembic migration → new router endpoint → new React component. AI needs all 5 steps.

📤

New export format

Add a service in backend/app/services/ that queries PostgreSQL and transforms the data. Wire it to a new router endpoint. The SDK CLI can then call it.

👥

New role / permission

Add the role string to the auth system in routers/auth.py, then add require_role("new_role") as a Depends on the endpoints it should protect.

🔔

New webhook event

Add to routers/webhooks.py. The webhook system already handles delivery, retries, and signature verification — you just define the event payload shape.
