Debug Like an Engineer

The most common failure modes, how to find them, and how to steer AI out of stuck loops

The usual suspects: what breaks most often

Every full-stack app has a predictable set of failure modes. Knowing these by name lets you tell AI exactly where to look — instead of watching it spin in circles for 10 minutes.

🚫
CORS errors

CORS fires when the frontend domain isn't in FastAPI's allowed origins list. Symptom: "Network Error" in browser console, request never reaches the backend. Fix: add your Vercel URL to allow_origins in backend/app/main.py.

🔐
JWT expiry — "suddenly logged out"

JWTs expire after 30 days. When they do, every API call returns 401. Symptom: user seems logged in (cookie exists) but all data requests fail. Fix: implement token refresh logic, or increase the expiry window for development.

🔌
Database connection pool exhausted

Under load, if too many async connections are held open, new requests time out. Symptom: 503 errors during traffic spikes. Fix: tune pool_size and max_overflow in the SQLAlchemy engine config.

📦
Stale Zustand state

Sometimes old annotation data persists in localStorage across sessions. Symptom: user sees tasks from a previous session, or their old answers auto-fill new tasks. Fix: call clearAnnotations() on new session start, or add a version key to the persist config.

🔄
Missing Alembic migration

AI adds a column to the model but forgets to migrate. Symptom: 500 errors when saving, with "column does not exist" in server logs. Fix: alembic revision --autogenerate -m "fix" then alembic upgrade head.

💡
The Async DB Gotcha

With async SQLAlchemy, you must await db.commit() to actually save to the database, and await db.refresh(annotation) to reload the record (with its database-assigned ID) from the DB. AI frequently writes the db.add() but forgets the commit. This is the #1 "data is not saving" bug in async FastAPI code.

Getting unstuck: break AI out of bug loops

AI gets into "debug loops" — trying the same fix 3 different ways when it doesn't understand the root cause. Here's how to break the loop using what you now know:

1
Name the layer, not just the symptom

Instead of "annotations aren't saving," say "the FastAPI route returns 200 but PostgreSQL never gets the write. I think the Alembic migration is missing or the await db.commit() is absent." This tells AI exactly where to look.

2
Read the server logs, not just the browser

The browser shows you the HTTP status code. The real error is in the FastAPI server logs (or Fly.io logs). Give AI the actual Python stack trace — it narrows the problem from 10 possibilities to 1.

3
Check Zustand state in browser devtools

Open browser devtools → Application → Local Storage → look for the Zustand key. If the annotation data is there, the problem is in submission (network/backend). If it's not there, the problem is in the UI (state management).

4
Isolate with a direct API call

Use the FastAPI automatic docs at /docs (built-in Swagger UI) to call the endpoint directly without the frontend. If it works there, the bug is in Next.js. If it fails there, the bug is in FastAPI or the database.

What to build next: the feature map

Now that you understand the architecture, here's how to think about extending it. Each feature lives in a predictable place:

📊

New annotation type

New Pydantic schema → new SQLAlchemy model → Alembic migration → new router endpoint → new React component. AI needs all 5 steps.

📤

New export format

Add a service in backend/app/services/ that queries PostgreSQL and transforms the data. Wire it to a new router endpoint. The SDK CLI can then call it.

👥

New role / permission

Add the role string to the auth system in routers/auth.py, then add require_role("new_role") as a Depends on the endpoints it should protect.

🔔

New webhook event

Add to routers/webhooks.py. The webhook system already handles delivery, retries, and signature verification — you just define the event payload shape.
