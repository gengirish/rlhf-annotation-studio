Meet the Cast

The five main characters in this codebase — what each one does, and why it exists

Five characters. One platform.

Think of this codebase like a film crew. Everyone has a specialized job. No one does everything. When you ask AI to "add a feature," knowing which character to give that job to is what separates a vibe coder from a great director.

⚛️
Next.js Frontend

The face of the app — everything in your browser. Built with React 18 + Zustand. Lives in frontend/. Handles all UI, navigation, and local state.

⚡
FastAPI Backend

The brain. A Python API server that handles authentication, data validation, and all business logic. Lives in backend/.

🗄️
PostgreSQL Database

The long-term memory. Every annotation, user, task, and export lives here permanently. FastAPI reads and writes to it using SQLAlchemy.

🔴
Redis

The speed cache. Some data (like session info) needs to be accessed in milliseconds, not seconds. Redis keeps hot data in memory for near-instant reads.

🐍
Python SDK + CLI

The power tools for technical users. The rlhf CLI lets you upload task packs, download datasets, and run evaluations from your terminal — no browser needed.

Where each character lives in the code

The project is a monorepo: one repository, multiple services. Here's the map:

frontend/ Next.js 14 app — everything users see
src/app/ Pages and routes (App Router)
src/lib/state/ Zustand store — your annotation workspace in memory
src/middleware.ts Guards protected pages — redirects if not logged in
backend/ FastAPI Python server
app/routers/ One file per feature: auth, tasks, reviews, datasets...
app/models/ SQLAlchemy database table definitions
app/services/ Business logic: LLM judge, certificates, exams, email
alembic/ Database migration scripts — how the schema evolves
sdk/ Python package — the rlhf CLI and HTTP client
docker/ Docker Compose — spins up Postgres + Redis locally
guidelines/ Human annotation rubrics (Markdown)

In production: two clouds, one app

When this app is deployed, it splits across two hosting providers — each playing to their strengths:

▲

Vercel — Frontend

Next.js pages are deployed to Vercel's global CDN. Users get fast page loads from a server near them, anywhere in the world.

🪰

Fly.io — Backend + Database

The FastAPI server and PostgreSQL database run on Fly.io, a platform that runs Docker containers close to users. The backend config lives in backend/fly.toml.

🔗
Why Split Them?

Vercel is unbeatable for static pages and SSR. Fly.io is better for long-running Python servers. Using the right tool for each job is a common production pattern — you'll hear engineers call this "separation of concerns."

A group chat: what happens when you log in

Here's what the components "say" to each other when you hit the login page:

🖥
Browser

User just typed their email + password and clicked Login.

⚛
Next.js

Got it. I'll POST those credentials to /api/v1/auth/login on the FastAPI server.

⚡
FastAPI

Request received. Let me check this password against the database...

🗄
PostgreSQL

Found the user. Their hashed password matches. Role is "annotator". Returning user record.

⚡
FastAPI

Password verified ✓. Generating a JWT token signed with the secret key. Valid for 30 days. Sending it back.

⚛
Next.js

Got the JWT. Storing it in localStorage AND setting an rlhf_session cookie. Redirecting to /dashboard.

🖥
Browser

Dashboard loaded! Every future request will include the JWT in the Authorization header.
